! Critical
No @Transactional — save and charge can desync
Lines 8, 14–15 · Data integrity · Distributed systems
Problem
orderRepository.save() commits to the database, then paymentService.charge() runs as a separate network call. If charge() throws — network timeout, gateway down, anything — the order row is already persisted with no payment attached. The system is now in an inconsistent state: a real order exists with no financial transaction.
Fix
Transactional Outbox Pattern Spring / JPA
@Transactional 
              
            @Service
            @RequiredArgsConstructor // Generates constructor for final fields (Lombok)
            public class OrderService {

                private final OrderRepository orderRepository;
                private final PaymentService paymentService;

                @Transactional
                public Order createOrder(String userId, List items) {
                    // 1. Validate Input
                    if (userId == null || items == null || items.isEmpty()) {
                        throw new IllegalArgumentException("Invalid order details");
                    }

                    // 2. Build Object
                    Order order = new Order();
                    order.setUserId(userId);
                    order.setItems(items);

                    // Use BigDecimal for money!
                    BigDecimal total = calculateTotal(items);
                    order.setTotal(total);

                    // 3. Persist and Charge
                    orderRepository.save(order);
                    paymentService.charge(userId, total);
                    // If this fails, @Transactional rolls back the save

                    return order;
                }

                private BigDecimal calculateTotal(List items) {
                    return items.stream()
                            .map(item -> item.getPrice()
                                    .multiply(BigDecimal.valueOf(item.getQuantity())))
                            .reduce(BigDecimal.ZERO, BigDecimal::add);
                }
            }
Impact
Without fix
Ghost orders — product shipped, revenue never captured. Reconciliation nightmare at scale.
Alternatives
Saga choreography, or at-least-once charge with idempotency key on the payment side.

Lines 20–21 · Financial accuracy · IEEE 754
Problem
IEEE 754 double cannot represent most decimal fractions exactly. 9.99 * 3 evaluates to 29.969999999999999, not 29.97. Accumulated across thousands of transactions these rounding errors compound into real monetary discrepancies — invisible until audited, and very hard to trace back.
Fix
Replace with BigDecimal Java
private BigDecimal calculateTotal(List<Item> items) { return items.stream() .map(i -> i.getPrice() .multiply(BigDecimal.valueOf(i.getQuantity()))) .reduce(BigDecimal.ZERO, BigDecimal::add); }
Impact
Without fix
Financial mismatches in ledgers, incorrect VAT totals, regulatory compliance failures.
Storage
Persist as DECIMAL(19,4) in DB. Never store money as FLOAT or DOUBLE.

Lines 3–6 · Testability · Spring best practice
Problem
Field injection uses reflection to bypass normal Java construction. Dependencies become invisible at the API level, preventing final fields and breaking standard unit test construction. The class is tightly coupled to the Spring container — impossible to instantiate without a full application context.
Fix
Constructor injection Spring / Lombok
@Service @RequiredArgsConstructor // Lombok generates the constructor public class OrderService { private final OrderRepository orderRepository; private final PaymentService paymentService; // explicit, immutable, testable }
Impact

Problem
createOrder(null, null) reaches the repository layer and produces either a NullPointerException deep in the stack or persists garbage data. Callers receive an unguarded 500 with a JPA stack trace instead of a meaningful 400 Bad Request. Validation belongs at the service boundary, not inside repositories.

public Order createOrder(@NotBlank String userId, @NotEmpty List<@Valid Item> items) { // Spring validates automatically if @Validated on the class Assert.hasText(userId, "userId must not be blank"); Assert.notEmpty(items, "order must contain at least one item"); ... }
Impact
Without fix

Fail fast at the boundary. Validate at the controller layer too, but always guard in the service.

~ Warning
No error handling on payment failure
Line 15 · Resilience · API contract
Problem
An uncaught RuntimeException from paymentService.charge() propagates uncontrolled. Callers receive a raw 500 with internal stack trace details. Payment failures are domain events — insufficient funds, card declined, gateway timeout — and each should produce a typed exception the caller can handle appropriately.
Fix
Domain exception wrapping Exception handling
try { paymentService.charge(userId, order.getTotal()); } catch (PaymentGatewayException ex) { log.warn("Payment failed for order {}: {}", order.getId(), ex.getMessage()); throw new PaymentFailedException( "Unable to process payment for order " + order.getId(), ex); }
Impact
Without fix
Raw traces leak to clients, incidents are hard to diagnose, retry logic impossible to implement cleanly.
Resilience
Pair with Resilience4j circuit breaker + retry for transient gateway failures.

Problem
A service that creates orders — a core business operation — emits zero telemetry.
